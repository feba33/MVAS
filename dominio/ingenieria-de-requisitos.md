# Ingeniería de requisitos

## Resumen
La **ingeniería de requisitos (RE)** es la disciplina que define, documenta y mantiene los requisitos de un sistema. Es la fase crítica del ciclo de vida del software (y de sistemas en general) que traduce necesidades de *stakeholders* en especificaciones verificables. Un mal RE es la causa principal de fallo de proyectos.

## Proceso (IESE / Elicitation)
1. **Elicitación**: entrevistas, talleres, observación, *personas*, *user stories*, *jobs-to-be-done*.
2. **Análisis y negociación**: resolver conflictos, priorizar (MoSCoW, Kano, WSJF).
3. **Especificación**: SRS (Software Requirements Specification), *user stories*, casos de uso, modelos.
4. **Validación y verificación**: revisión, prototipos, *acceptance criteria*.
5. **Gestión del cambio**: trazabilidad y control de versiones de requisitos.

## Tipos de requisitos
- **Funcionales** vs **no funcionales** (NFR: rendimiento, seguridad, usabilidad — ISO/IEC 25010).
- **De usuario** vs **del sistema**; **emisión/restricción**.
- Requisitos **regulatorios** (cumplimiento).

## Estándares
- **ISO/IEC/IEEE 29148**: RE y SRS.
- **IEEE 830** (histórico). Gestión de requisitos con herramientas (DOORS, Jira, Polarion).
